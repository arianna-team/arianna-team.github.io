Introduction

ARIANNA supports device manufacturers with an efficient and scalable vulnerability management process across the full product lifecycle. The platform combines Device Model Management, continuous vulnerability monitoring and reporting, and workflow-based triage and prioritization.

Core capabilities include:

  • HBOM + SBOM modeling for hardware and software visibility.
  • Daily correlation with vulnerability intelligence sources.
  • End-to-end status tracking from discovery to closure/disclosure.
  • Audit-ready evidence and exportable reports.

Vulnerability Management Project

A Vulnerability Management Project is the operational container for one product/device/system. It allows teams to create Device Models, review vulnerability reports, perform triage, update risk and status, and track remediation over time.

Projects are designed for continuous collaboration across security, engineering, and compliance stakeholders.

Vulnerability Management Process

1. Identification

Collect newly disclosed vulnerabilities from monitored sources and map them to relevant components.

2. Triage

Assess applicability, device-context risk, and remediation priority. Outcomes include applicable, not applicable, or accepted risk.

3. Addressing

Apply updates, patches, removals, or mitigations depending on feasibility and operational impact.

4. Closure and Disclosure

Finalize status, preserve action traceability, and communicate required information internally/externally.

Device Model

The Device Model is the full component inventory (HBOM + SBOM), organized by Processing Units and Groups. ARIANNA emphasizes hardware context because exploitability and remediation options can depend on architecture, interfaces, and update mechanisms.

See also Device Model lifecycle management.

Processing Units

A Processing Unit (PU) is a logical hardware block (MCU, MPU, memory, secure element, hardware module) used to structure component and vulnerability analysis.

Each PU should include: name, description, type, system type, vendor/part number, interfaces, and component set.

Components and Groups

Components represent monitored hardware or software items and are the basis for vulnerability correlation.

Groups aggregate components by layer/function (hardware, firmware, OS, application) to simplify triage and reporting.

Getting Started

Device Description (initial)

Identify all PUs and document execution environments and interfaces. This stage seeds HBOM generation and later vulnerability context.

Device Model Creation (per release)

HBOM typically stays stable unless hardware changes. SBOM is updated for each firmware/software release. ARIANNA tools assist accurate SBOM extraction and normalization.

Software Composition Analysis (SCA) Tools

ARIANNA provides environment-specific SCA scripts (Python/Bash/PowerShell) that run in build/dev environments and output component data in CSV for platform upload.

  • No source code disclosure required.
  • No reverse engineering required.
  • No agent installation on target device.

Supported environments include OS distributions, application stacks (.NET/Python/PHP/JavaScript), MCU SDKs, and cloud/container contexts.

Vulnerability Reporting

ARIANNA generates daily reports and enriches each vulnerability with severity, exploitability, KEV, EPSS, CWE, and lifecycle status.

Report sections include:

  • Overview and statistics (open/closed trends, risk trends, deadlines).
  • Device Model view (component-centric analysis).
  • Vulnerability list (workflow-centric view).
  • Highlights and prioritization cues.

Device Model Lifecycle

ARIANNA tracks multiple model versions over time. On new releases, component changes can trigger new vulnerability notifications or automatic closures where affected components are updated/removed.

Recommended versioning strategies: semantic versioning or calendar versioning.

Triage and Prioritization

The pre-triage engine supports high/low priority classification based on configured policy criteria:

  • Risk-based: severity/risk class.
  • Exploitability-based: KEV membership, EPSS thresholds, exploit maturity, attack vector realism.

ARIANNA supports decision-making, but final risk acceptance and remediation choices remain with the security team.

Policy and Deadlines

Organizations can configure remediation timeframes by risk level and monitor Expiring/Expired items. This enables policy enforcement and supports regulatory compliance objectives.

Exports and Auditing

Reports can be exported in machine-readable formats (CycloneDX, SPDX, VEX) and human-readable PDF outputs for audits, customer communication, and internal governance.

CI/CD Integration

Through REST APIs, pipelines can create Device Model versions, upload SCA outputs, trigger report generation, and retrieve filtered high-priority vulnerabilities.

Users and Roles

ARIANNA defines System Administrator and Regular User roles (Project Owner, Member, Observer) with scoped permissions for administration, triage actions, and read-only access.

Password and Access Security

Password policy includes minimum complexity requirements (length, uppercase/lowercase/digit/special character). Password recovery is handled with email verification and secure reset flow.
